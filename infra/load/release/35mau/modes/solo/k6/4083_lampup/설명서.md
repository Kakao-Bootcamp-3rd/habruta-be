1. 필요성
Habruta은 사용자가 말하기 연습을 하고 AI 피드백을 받는 교육 서비스로, 이용 시간이 퇴근·하교 이후인 20시~24시에 집중될 가능성이 높다. 따라서 단순 평균 트래픽이 아니라, 사용자가 몰리는 피크 시간대에 서비스가 안정적으로 동작하는지 검증할 필요가 있었다. 이를 위해 목표 피크 동시 접속자 4,083명을 기준으로 단계적 램프업 부하 테스트를 설계하였다.

2. 얻으려고하는게 뭔지
이번 테스트의 목적은 단순히 서버가 “버티는지”만 확인하는 것이 아니라, 동시 접속자가 증가할 때 어느 구간부터 응답 지연이나 병목이 발생하는지 확인하는 것이다.

확인하려는 지표는 다음과 같다.

| 확인 항목         | 목적                  |
| ------------- | ------------------- |
| p95 응답 시간     | 사용자가 체감하는 지연 확인     |
| 요청 실패율        | 대량 요청 시 오류 발생 여부 확인 |
| TPS/RPS       | 목표 트래픽 처리량 충족 여부 확인 |
| CPU/메모리 사용률   | 애플리케이션 서버 병목 확인     |
| DB 커넥션/쿼리 지연  | 데이터베이스 병목 확인        |
| Redis 지연/타임아웃 | 캐시 및 실시간 기능 병목 확인   |



3. 진행과정

| 단계  |          부하 수준 |     시간 | 목적                         |
| --- | -------------: | -----: | -------------------------- |
| 1단계 |     0 → 1,000명 |     5분 | 초기 유입 및 서버 예열 구간 확인        |
| 2단계 | 1,000 → 3,000명 |    10분 | 피크 대비 약 70% 수준에서 병목 징후 확인  |
| 3단계 | 3,000 → 4,083명 |    10분 | 목표 피크 동시 접속자까지 확장 가능 여부 확인 |
| 4단계 |      4,083명 유지 | 10~20분 | 피크 부하 지속 상황에서 안정성 검증       |
| 5단계 |     4,083 → 0명 |   3~5분 | 부하 제거 후 자원 회복 상태 확인        |



4. 결과
테스트 결과, 동시 접속자가 증가함에 따라 p95 응답 시간, 요청 실패율, 서버 CPU, DB 커넥션, Redis 응답 지연 등을 비교하여 병목 발생 구간을 확인하였다.
| 구간        | p95 응답 시간 | 실패율 | 주요 관찰 결과                 |
| --------- | --------: | --: | ------------------------ |
| 1,000명    |           |     | 기본 요청 처리 안정성 확인          |
| 3,000명    |           |     | 피크 대비 70% 구간에서 병목 여부 확인  |
| 4,083명    |           |     | 목표 피크 동시 접속자 처리 가능 여부 확인 |
| 4,083명 유지 |           |     | 지속 부하 상태에서 안정성 확인        |
| Ramp-down |           |     | 부하 제거 후 자원 회복 여부 확인      |
